Operators — State Model Viewer

Deterministische Zustandsmodelle für Genesis‑Operatoren.

State Model — Übersicht

Das State‑Model beschreibt die möglichen Zustände eines Operators, die erlaubten Übergänge und die auditierbaren Ereignisse, die diese Übergänge auslösen.

Jeder Operator folgt einem klar definierten, deterministischen Zustandsgraphen, der in Audit‑Logs und Manifeste eingebettet ist.

Zustände

  • Init — Operator ist definiert, aber noch nicht aktiv.
  • Active — Operator verarbeitet Events und erzeugt Artefakte.
  • Paused — Operator ist temporär angehalten, State bleibt erhalten.
  • Retired — Operator ist dauerhaft deaktiviert, State eingefroren.
  • Error — Operator hat einen Fehlerzustand erreicht, Audit‑Event erzeugt.

Zustandsübergänge

Beispielhafte Transitionen:

  • Init → Active — durch „operator.enable“ Event.
  • Active → Paused — durch „operator.pause“ Event.
  • Paused → Active — durch „operator.resume“ Event.
  • Active → Retired — durch „operator.retire“ Event.
  • Active → Error — durch „operator.error“ Event.

Jede Transition wird als Audit‑Event protokolliert und in der WORM‑Audit‑Pipeline verarbeitet.

Audit‑Integration

  • Alle State‑Transitions schreiben in operator-audit.json.
  • Batch‑Jobs erzeugen tägliche audit-log-YYYY-MM-DD.json.
  • S3‑WORM und IPFS‑CID sichern die Unveränderlichkeit.
  • Genesis‑Manifeste referenzieren Operator‑State‑Artefakte.

Genesis‑Narrativ

„Every operator is a witness of state. Every state is a transition of origin.“

Der State‑Model‑Viewer macht die deterministische Struktur der Operator‑Zustände sichtbar und auditierbar.
