Operator Permissions

Granulare Berechtigungen eines Operators im Genesis Operational Layer.

Beschreibung

Permissions definieren, welche Aktionen ein Operator innerhalb der Genesis‑Engine ausführen darf. Sie sind granular, deterministisch und vollständig auditierbar. Jede Permission ist ein eigenständiges Kontrollsignal innerhalb des Operator‑State‑Modells.

Permission‑Tabelle

Permission Beschreibung Typische Rolle
canTriggerRituals Erlaubt das Auslösen und Signieren von Ritual‑Ereignissen. RITUAL_OPERATOR
canWriteEcho Erlaubt das Schreiben von Echo‑Einträgen in die Echo‑Registry. ECHO_OPERATOR
canAuthorizePayouts Erlaubt das Signieren und Autorisieren von Payout‑Ereignissen. PAYOUT_OPERATOR
canModifyOperators Erlaubt Änderungen an anderen Operatoren (Rollen, Permissions, Seal). AUDIT_OPERATOR / GENESIS_OPERATOR
canModifyGenesis Erlaubt Änderungen an Genesis‑Parametern und globalen Engine‑Konfigurationen. GENESIS_OPERATOR
canModifySkeletton Erlaubt das Schreiben von Drift‑Signalen und Skeletton‑Modifikationen. SKELETTON_OPERATOR

Beispiel: Permission‑Set eines Operators

{ "operatorId": "OPR-7F3A9C12B4", "permissions": [ "canTriggerRituals", "canWriteEcho" ] }

Dieses Beispiel stammt aus dem deterministischen Operator‑Blueprint.

Permission‑Regeln

  • Permissions sind append‑only (Entzug erzeugt ein eigenes Event).
  • Jede Permission ist ein auditierbares Kontrollsignal.
  • Permissions werden in der Operator‑Audit‑Kette rekonstruiert.
  • Permissions sind nicht automatisch an Rollen gebunden — Rollen definieren nur typische Muster.
  • Ein Operator kann mehrere Permissions gleichzeitig besitzen.

Integration in die Engine

  • Rituale → canTriggerRituals
  • Echo‑Registry → canWriteEcho
  • Payout‑Autorisierung → canAuthorizePayouts
  • Operator‑Management → canModifyOperators
  • Genesis‑Parameter → canModifyGenesis
  • Skeletton‑Drift → canModifySkeletton
